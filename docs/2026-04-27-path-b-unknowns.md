# Path B: critical unknowns research — 2026-04-27

## Verdict

**CONDITIONAL GO** on Phase 1 commitment, with a hard scope adjustment: Phase 1 must explicitly drop "any feature that requires authoritative server-side combat/AI in the same milestone." Avledet today is a **ZDO replication and ownership broker**, not a simulation engine. Three of the four unknowns (raycast, animation timing, simulation tick) are *not partially built* — they are *not built at all*. Pathfinding has the data layer (heightmap exists in C++) but neither the API exposure nor the placed-structure occupancy data needed for true mob nav. A six-month milestone that only does ownership reclaim + idle-walk stub mobs is feasible. A milestone that includes "mobs hit players" is not.

## 1. Raycasts

**What exists:** Nothing. `library/include/VUtilsPhysics.h` declares only `PointInsideRect`, `RectInsideRect`, `RectOverlapRect` (marked `[[deprecated("broken")]]` at line 24-30), `BoxBoxOverlap` (a clean OBB-vs-OBB SAT test, lines 32-34), and quaternion conversion helpers. `VUtilsPhysics.cpp:306-391` implements `BoxBoxOverlap` correctly but there is **no ray, no segment, no swept volume, no spatial broadphase**. The single literal mention of "Raycast" in the entire C++ source is a Unity-port stub left behind in `library/src/Heightmap.cpp:661-665`:

```cpp
// Ray ray = new Ray(point + Vector3f::UP * 100.f, Vector3f::DOWN);
// RaycastHit raycastHit;
// if (this->m_collider.Raycast(ray, raycastHit, 300)) { ... }
return -10000;
```

The function it lives in (`Heightmap::GroundHeight`) is dead — `assert(false)` at line 659 — because Avledet uses analytical heightmap sampling (`GetWorldHeight`, `GetHeight(x,y)` at `HeightMap.h:152-156`) instead of physics ground probes.

**What's missing for combat:**
1. **Heightmap raycast.** Needed for "is target below me / behind a hill". Heightmap is a 65×65 vertex grid per zone (`E_WIDTH = UNITS_PER_ZONE + 1 = 65`, `HeightMap.h:67`, `ZoneManager.h:194`), 1m vertex spacing. A naive DDA-style 2.5D ray walk against the height grid is 50–100 LOC of C++ once `HeightmapManager::GetHeightmap(point)` is exposed.
2. **ZDO/structure raycast.** Player-placed walls, doors, etc. live as ZDOs but only carry position/rotation/prefab — **no collider geometry server-side**. Prefab dimensions live in the client-only Unity assets. To raycast against structures, Avledet would need a prefab → AABB/OBB lookup table (data not present in Avledet today).
3. **Lua exposure.** Zero physics primitives are exposed to Lua. `APIZone.cpp` exposes `IZoneManager` but **no `IHeightmapManager`** at all (verified — `grep Heightmap library/src/API*.cpp` returns nothing). `BoxBoxOverlap` from `VUtilsPhysics` is also unexposed.

**LOC estimate to fill the gap (C++ side):**
- Expose `HeightmapManager:get_world_height(pos) -> float`: ~10 LOC.
- DDA heightmap raycast in C++: ~80 LOC.
- Prefab AABB table (hand-curated or imported from Valheim Asset Bundle dump): ~200 LOC + a several-hundred-row data table.
- ZDO broadphase (zone-bucketed AABB sweep): ~150 LOC, leveraging the existing `m_objectsBySector` array (`ZDOManager.h:42-45`).
- Lua bindings: ~30 LOC across `APIZone.cpp` / new `APIPhysics.cpp`.

**Total: ~470 LOC + a prefab dimension dataset.** The dataset is the critical-path item — without it, mobs cannot tell a wall from open ground.

**Cited files:** `library/include/VUtilsPhysics.h:1-44`, `library/src/VUtilsPhysics.cpp:306-391`, `library/src/Heightmap.cpp:656-667`, `library/include/HeightMap.h:67,152-156`, `library/src/API.cpp:32-41`, `library/src/APIZone.cpp:51-62`.

## 2. Pathfinding

**Heightmap layout:** 64m × 64m zones, 65×65 vertex grid per zone (1m spacing), height as `float`, biome corners stored as 4-element array, paint mask as RGBA color per vertex (`HeightMap.h:67-80,151-161`). World radius is 10500m (`GeoManager.h:243`), inner-zone count is `WORLD_INNER_ZDIAMETER² ≈ 328²` zones. Heightmaps are lazily generated by `HeightmapBuilder` on a worker thread (`HeightmapBuilder.h:15-50`) and cached in `IHeightmapManager::m_heightmaps` (`HeightmapManager.h:13`). Water/ocean is not stored as voxels — it's an analytical height check (`GetOceanDepth` at `HeightMap.h:121`); below sea level = water. Lava (Ashlands) similarly handled by biome height functions in `GeoManager`.

**A* search-space size for 100m mob travel:** At 1m grid resolution that's 100×100 = 10k cells worst-case open-list, well within budget for a single mob path computed once per few seconds. With diagonal-allowed Manhattan heuristic, expected expanded nodes ≈ 200–800 for typical terrain. Lua A* at ~5μs per node = 4ms per path. **Per-frame cost is fine; total cost of 50+ concurrent mobs pathing simultaneously is what would hurt.** Mitigation: stagger pathfinder ticks (1–2 mobs per frame) and cache subgraphs.

**Walls / placed structures:** Player-placed structures are ZDOs in the spatial sector index (`ZDOManager.h:42-45`, `m_objectsBySector` indexed by 64m zones). The pathfinder *can* enumerate them via `IZDOManager::GetZDOs(pos, radius)` (`ZDOManager.h:323-326`) and `SomeZDOs` (`ZDOManager.h:249-256`), both already Lua-exposed. **However, structure footprints are unknown** without the prefab AABB table from §1 — the ZDO only carries pos/rot/prefab_hash. So the pathfinder either treats every structure ZDO as a hard 1m blocker (cheap, terrible UX — mobs walk through walls or around 1m posts inappropriately) or relies on the prefab dimension dataset.

**Water/lava avoidance:** Trivially queryable per-vertex via `Heightmap::GetWorldHeight` + sea level constant. Biome lookup (`Heightmap::GetBiome` at `HeightMap.h:124`) classifies Ashlands lava regions. Both currently unexposed to Lua.

**LOC for basic A* in Lua:** ~150 LOC for vanilla A* with binary heap open-list, plus ~50 LOC for the heightmap-grid adapter. Realistic working module: **200 LOC of Lua + the C++ heightmap-exposure work from §1**.

**Cited files:** `library/include/HeightMap.h:33-170`, `library/include/HeightmapManager.h:10-51`, `library/include/HeightmapBuilder.h:15-50`, `library/include/ZoneManager.h:194-206`, `library/include/ZDOManager.h:42-45,249-256,323-326`, `library/include/GeoManager.h:79,143-157,222,243`.

## 3. Animation timing

**What Avledet has:** Animation states **as data**, not **as timing**. `Hashes.h:788-809` defines `ZSyncAnimation` field hashes (`ALERT`, `ATTACH_BED`, `BLOCKING`, `CRAFTING`, `EQUIPPING`, `FLAPPING`, `FORWARD_SPEED`, `STATE_F`, `STATE_I`, ...) which clients write into ZDOs as the animator state machine progresses. `Hashes.h:822-1399` enumerates hundreds of attack-animation hashes (`Abomination_attack1`, `GoblinBrute_RageAttack`, `fx_jotunbane_swing` etc.) — these are pure identifiers used in RPC events, not timed simulation handles.

**Combat RPC handling today:** Searched `library/src/` for damage RPC registration (`Register(.*Damage|Hit|ApplyDamage)` against `RouteManager.cpp`, `Avledet.cpp`, `NetManager.cpp`): **zero registrations**. The combat hashes (`Hashes.h:182,208,261,265,300,354,360,370`) are constants only — Avledet does **not** intercept, validate, or handle damage RPCs. They are routed through `IRouteManager::InvokeView`/`Invoke` (`RouteManager.h:67-180`) as opaque payloads from one client to the ZDO's owner peer. The owner peer's *client* runs `Character::Damage` etc.; Avledet just forwards bytes.

**Server knowing "this attack will hit at T+0.4s":** Not without simulating. Vanilla Valheim wraps attack windups in client-side animator state; the server (vanilla or Avledet) only sees "client says: I attacked, dealt N damage to ZDO X" *after the fact*. The RPC arrives post-hit. There is no pre-hit channel.

**Approach options:**
1. **Pure observer (recommended for Phase 1):** Subscribe to the `'RouteInAll'` event on the damage hashes from Lua, log/sanity-check, but do not author hits. This is what `ServerAuthority` C# already does as Phase 1 anti-cheat.
2. **Server-authoritative, anim-blind:** Implement combat as cooldown timers + range checks ignoring animation. Mob "swings" by triggering a delayed Lua task that does the hit at T+0.4s (cooldown lifted from a hard-coded per-attack table). Animation hash sent to clients via `RouteManager:invoke_all` for visual sync — but timing is server-driven and **will mismatch animation visuals by 50–200ms** because clients lerp animator state over multiple frames. Acceptable for a stub, ugly for production.
3. **Full anim-aware:** Port Valheim's `ZSyncAnimation` state machine to C++. Multi-month effort. Out of scope for any reasonable Phase 1.

**Recommendation: option 1 for Phase 1, option 2 for Phase 2 mobs.** Authoritative combat is a Phase 3+ effort.

**Cited files:** `library/include/Hashes.h:118,182,208,261,265,300,354,788-809,822-1399`, `library/include/RouteManager.h:67-180`, `library/src/Avledet.cpp:892-924` (main update has no combat tick), `library/src/API.cpp:167-173` (RouteManager Lua surface).

## 4. Client prediction reconciliation

**Current state:** Avledet uses Valheim's native `(owner_rev, data_rev)` pair to gate ZDO sync (`ZDOManager.cpp:918-930`, `Peer.cpp:247-254`):

```cpp
return find == m_zdos.end()
    || zdo->get_owner_rev() > find->second.first.get_owner_rev()
    || zdo->get_data_rev()  > find->second.first.get_data_rev();
```

`SendZDOs` sends a **full ZDO snapshot** per outdated ZDO — `zdo->pack(writer, true)` at `ZDOManager.cpp:927`, not a field-level diff. So delta is *per-ZDO* (skip if peer's revision is current), not *per-field*. Unchanged ZDOs are skipped entirely; changed ZDOs re-send the entire packed payload (~50–500 bytes each).

**Reconciliation pattern in bundled Lua:** `data/lua/scripts/Portals/PortalsZDO.lua:32-42` shows the canonical pattern: take ownership (`portal.owner = Avledet.id`), mutate ZDO state, then `ZDOManager:force_send_zdo(portal.id)` to bump revision and push. Or, if not stealing ownership, send `RouteManager:invoke(portal.owner, RPC_…)` to ask the owning client to mutate. **There is no client/server position reconciliation pattern, because Avledet has no notion of "server position vs client position" — there is only "ZDO position", authored by whoever has ownership.** The owning peer is the source of truth.

**Vanilla Valheim prediction:** Searched the decompiled `ZDOMan.cs`: zero matches for `interpolate|predict|smooth|reconcile|rubber`. Vanilla Valheim does not do client-side prediction in the FPS sense. Each character's ZDO is *owned* by exactly one peer (its controller), and that peer is authoritative for position. Other peers receive ZDO updates and lerp visually toward the new position — no rollback, no reconciliation, no rubber-banding (because there's no authority disagreement to reconcile).

**Risk assessment:** **Low for Path B Phase 1, high for Phase 3+.**
- Phase 1 (ownership reclaim, idle-walk mobs): server takes ownership → server is the source of truth → no divergence possible.
- Phase 3 (server-authoritative combat): if we move player-character ownership to the server, we'd have to *invent* prediction/reconciliation from scratch on the **client** side — and Valheim clients don't ship with that capability. The vanilla client expects to own its character ZDO. Forcing server ownership of player ZDOs would cause every player input to roundtrip through the server before the local player sees movement. That's a core Valheim-client-rewrite problem and **the path-B approach cannot solve it without modding the client** — which removes Path B's advantage over `ServerAuthority` mod.

**This is the most important architectural finding of the session.** "Server-authoritative Valheim" with vanilla clients can authoritatively own non-player entities (mobs, structures, projectiles in flight) but **cannot meaningfully take authority for player movement** without bespoke client work. Path B's MMO claim must be scoped to "authoritative world / mobs / structures, client-authoritative players" — which is exactly what vanilla Valheim already does.

## Recommended Phase 1 scope adjustments

The current Phase 1 stub plan ("random idle-walk, no combat") is **viable as written** but should add three explicit non-goals:

1. **No raycasting in Phase 1.** Mobs walk in straight lines or cardinal-direction wanders, no LOS checks, no obstacle avoidance beyond not falling into water. Acceptable for a smoke test of "Avledet runs Lua mob loops at 50-mob scale".
2. **No A* in Phase 1.** A* requires the C++ heightmap-exposure work (~470 LOC) which is itself a milestone. Move it to Phase 2.
3. **No damage in Phase 1.** Mobs are decorative. Players cannot kill them, they cannot kill players. This is honest about what Avledet actually provides; "stub combat" is a worse lie than "no combat".

**Add to Phase 1 (small, but they unblock Phase 2):**
- Spike: instrument `IAvledet::update()` cadence — measure actual tick time at idle and with 50 mobs running a 5Hz Lua wander loop. We have not measured this and it's the load-bearing assumption for everything downstream.
- Spike: implement and Lua-expose `IHeightmapManager` (~50 LOC C++ + ~10 LOC `APIHeightmap.cpp`). Cheap, unblocks everything in Phase 2.
- Spike: instrument prefab AABB extraction. Decide: hand-curate top-30 prefabs, or batch-extract from Valheim Unity assets.

**Move to Phase 2:** raycast, A*, structure-aware pathing.
**Move to Phase 3 (or postpone):** server-authoritative combat, anim-aware hits, projectile arcs.
**Strike from roadmap:** server-authoritative *player* movement. Architecturally infeasible without a custom client.
